# 個人記帳系統 PRD (產品需求文件)

## 1. 專案概述
- **背景與動機**：現代人生活忙碌，經常難以確實掌握每天的花費與收入狀況。為了能更有效地管理個人財務，避免不必要的超支，我們需要一個簡單且直覺的記帳系統。
- **目標用戶**：有日常記帳需求、希望確實掌握收支狀況與剩餘金額的個人使用者。
- **核心價值主張**：提供輕量快速的記帳體驗，讓使用者可以輕鬆地記錄收支並隨時查看目前的總結餘。

## 2. 功能需求

### 2.1 新增收支紀錄
- **使用者故事**：作為個人使用者，我希望能夠新增支出與收入紀錄（輸入包含金額、類別及備註），以便我詳細且準確地紀錄每一筆財務變動。

### 2.2 檢視歷史帳目清單
- **使用者故事**：作為個人使用者，我希望能在首頁或特定頁面查看所有的歷史帳目清單，以便我回顧過去的花費項目。

### 2.3 自動計算總餘額
- **使用者故事**：作為個人使用者，我希望系統能自動根據所有已輸入的歷史收支紀錄，計算出現在的總餘額，以便我隨時了解手邊的可用資金。

### 2.4 刪除錯誤紀錄
- **使用者故事**：作為個人使用者，我希望能夠刪除輸入錯誤或者是多記的帳目紀錄，以便維持記帳資料的正確性。

### 2.5 類別分類與篩選（補足第五項必備功能）
- **使用者故事**：作為個人使用者，我希望能夠針對我的紀錄選擇對應的「類別」，並且系統能夠依照不同類別歸納顯示，以便我了解各個類別的花費狀況。

## 3. 非功能需求
- **技術限制**：本專案必須使用 Flask 作為開發後端框架，搭配 Jinja2 處理前端頁面的打版渲染，並以輕量級關聯式資料庫 SQLite 作為儲存方案。
- **效能與安全考量**：
  - 確保使用者輸入的資料有進行基礎消毒，防止 XSS 攻擊。
  - 使用參數化查詢（Parameterized Query）來存取 SQLite 資料庫，避免 SQL Injection。
  - 整體操作（新增、刪除及載入首頁）需能在短時間內（< 1秒）反應，提供使用者流暢的體驗。

## 4. MVP 範圍 (最小可行性產品)

### Must Have（必須具備）
- 能夠建立包含金額、收支屬性、類別與備註的紀錄。
- 列出所有的歷史收支清單。
- 介面上清楚顯示自動計算後的累計總餘額。
- 提供刪除特定紀錄的功能。

### Should Have（應該具備）
- 內建數種常用的預設收支類別（例如：飲食、交通、娛樂、薪水、其他）以加速使用者輸入。
- 顯示紀錄建立的日期。

### Nice to Have（有了更好）
- 能夠修改/編輯已送出的紀錄（而不僅僅只能刪除後再新建一筆）。
- 依日期（月份）或類別對帳目進行篩選、呈現圓餅圖表分析功能。
